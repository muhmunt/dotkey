# Docker & Docker Compose

## Pattern: pull before docker-compose up

Generate a `.env` file before starting your containers:

```bash
dotkey pull production --project my-service
docker-compose up -d
```

Your `docker-compose.yml` uses the file:

```yaml
services:
  app:
    image: my-app
    env_file:
      - .env   # generated by dotkey pull
```

## Makefile workflow

```makefile
.PHONY: dev prod

.env:
	dotkey pull development --project my-service

dev: .env
	docker-compose up

prod:
	dotkey pull production --project my-service
	docker-compose -f docker-compose.prod.yml up -d
```

## CI with Docker

```yaml
- name: Pull secrets
  run: dotkey pull production --project my-service
  env:
    DOTKEY_TOKEN: ${{ secrets.DOTKEY_TOKEN }}
    DOTKEY_API_URL: https://your-dotkey-instance.com

- name: Build and run container
  run: |
    docker build -t my-app .
    docker run --env-file .env my-app
```

## direnv (.envrc)

Auto-load your `.env` when you `cd` into the project:

```bash
# .envrc — safe to commit, loads the .env file
dotenv_if_exists .env
```

Then refresh secrets:

```bash
dotkey pull development --project my-service
direnv reload
```

## Tips

- Add `.env` to `.gitignore` — it contains secrets
- Commit `.envrc` and `.dotkey` — they contain no secrets
- Add `dotkey pull` as a pre-start step in your local dev scripts
